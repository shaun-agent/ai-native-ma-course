#5. LLM, Harness, Tool Calling, Agent

#Key Line

An LLM is a probabilistic next-token model trained at scale. Its output becomes stochastic when the decoding or product layer samples from those probabilities. It is powerful because many human tasks can be represented as text, code, symbols, and examples. Everything that supports or changes the behavior of a raw LLM is a harness: context, tools, memory, permissions, feedback, guardrails, process design, and stop conditions. An agent is an LLM operating inside a harness toward a goal.

#Why This Section Matters

We often say:

Agent = LLM + Harness

This is elegant, but needs clarification:

  • Many current LLM products already include a built-in harness.
  • ChatGPT, Claude, Gemini, Cursor, Codex, and similar systems are not "raw models."
  • They usually include tool access, memory/context windows, file handling, runtime permissions, UI constraints, safety policies, and hidden orchestration.
  • So when we build agents, we are often building another harness around an already-harnessed model.

Better teaching sentence:

Agent = model capability + context + tools + control loop + permissions + verification.

#Deterministic Vs Stochastic Output

Start with this contrast before defining LLM.

Deterministic system:
same input + same state + same rules = same output

Stochastic system:
same input can produce different outputs because the system samples from probabilities or uses randomness internally

Concrete deterministic examples:

  • Calculator: 123 + 456 = 579 every time.
  • SQL query: same deterministic query + same database state = same result; visible ordering requires ORDER BY.
  • Sorting function: same list + same sort rule + deterministic tie handling = same order.
  • Rule engine: if invoice > 100000, require approval always triggers under the same condition.
  • Unit test: same code + same test environment should pass or fail the same way.

Concrete stochastic examples:

  • LLM generation with sampling: same prompt can produce three different summaries.
  • Image generation without a fixed seed: same prompt can create different images.
  • K-means training without fixed initialization: same data can lead to different centroids.
  • Online recommender / ad system with exploration: the scoring model may be deterministic, but the product may intentionally sample, explore, or change ranking as context changes.

Key teaching point:

Deterministic systems are easier to verify. Stochastic systems can be more flexible, but need harness, tests, review, and audit.

Important nuance:

  • Greedy decoding or temperature near zero can make an LLM more repeatable.
  • Sampling controls such as temperature and top-p affect how much randomness enters generation.
  • This does not turn the LLM into a normal deterministic rules engine.
  • Even when output is repeatable, the model is still producing tokens from learned probability structure, not executing a human-readable rulebook.

#Foundation Note: Are ML Models Deterministic?

Short answer:

Many common ML models are deterministic at inference time, but ML as a whole is not always deterministic. LLMs are a subtype of ML, and many LLM products expose stochastic generation as a core behavior.

Separate three layers:

Training:
often stochastic

Model inference:
often deterministic if weights + input + runtime + decoding rule + random seed / deterministic kernels are fixed

Product/system output:
may still be stochastic because of sampling, retrieval, tools, model routing, or changing context

Concrete examples:

  • Logistic regression classifier: once trained, same features usually produce the same probability.
  • Decision tree / random forest: once trained, same input usually produces the same label or score.
  • Neural network classifier: with dropout off and fixed weights, same image usually produces the same class probabilities.
  • K-means: training may vary because random initialization can choose different clusters; after centroids are fixed, assignment is deterministic.
  • Recommender system: the scoring model may be deterministic, but the product ranking can change because inventory, user behavior, exploration, and business rules change.
  • LLM: the forward pass can deterministically produce a next-token probability distribution under fixed conditions; with sampling, the same prompt can produce different valid continuations.

Whiteboard correction:

ML model != always stochastic
ML system != always deterministic
LLM = ML subtype with generative token prediction

Use this wording:

The model's forward pass can be deterministic as a mathematical function, while the training process, decoding layer, and AI product can still be stochastic as a system.

Why this matters for AI Builders:

  • If output must be exact, use deterministic rules, calculators, validators, tests, or database queries.
  • If output can be drafted, interpreted, ranked, summarized, or explored, stochastic generation may be useful.
  • Production AI often combines both: probabilistic or stochastic model output plus deterministic checks.

#What Defines An LLM

LLM means Large Language Model.

Definition for the whiteboard:

A large neural network trained on massive text/code/token sequences to predict a probability distribution over likely next tokens, then used through context to perform many language-representable tasks.

What defines it:

  • It operates on tokens, not directly on "truth."
  • It learns statistical structure from huge corpora.
  • Common autoregressive LLMs generate one token after another.
  • It can use context at inference time: instructions, examples, documents, prior messages, tool results.
  • It is general-purpose because many tasks can be represented as language, code, schemas, or symbols.

What does not define it:

  • It is not a database.
  • It is not a search engine by itself.
  • It is not a calculator by itself.
  • It is not a rules engine.
  • It is not a workflow engine.
  • It is not a truth oracle.
  • It is not the owner of product intent.

Concrete example:

Question: "What is our refund policy?"

Raw LLM:
predicts a plausible answer from learned patterns and prompt context

LLM + retrieval harness:
reads the actual policy doc, quotes the relevant section, drafts an answer

LLM + workflow harness:
checks order status, applies refund rules, asks for approval if risk is high

The same model becomes more reliable when the harness supplies source material, tools, rules, and verification.

#Harness: Broad Definition

Use this broad definition on the whiteboard:

Harness = everything around the raw LLM that shapes behavior

That includes:

  • context package
  • system prompt / instruction hierarchy
  • examples and rubrics
  • reasoning or thinking-process design
  • tool definitions and schemas
  • memory and retrieval
  • workspace permissions
  • guardrails
  • human approval gates
  • tests, evals, and review protocols
  • logs and audit trail
  • UI / product constraints

Critical point:

Guardrail is harness. Thinking-process design is harness. Tool permission is harness. Review protocol is harness.

This is why AI Builders should not only ask, "Which model is bigger?"

They should ask:

What harness makes this model reliably useful here?

#Why Harness Engineering Changes Capability

Scaling improved raw model capability. But many real-world jumps now come from the system around the model:

raw model scale
  + context window
  + tools
  + reasoning process
  + guardrails
  + retrieval
  + sandbox
  + evals
  + human review
  = new practical capability border

This is not "defeating" scaling laws in the research sense. It means product capability can move beyond what raw parameter count alone would suggest.

#Older Definition Of Agent

Older / simpler definition:

An agent is an LLM system that can call tools.

This was useful because tool calling separated agents from pure chatbots:

Chatbot: answer in text
Agent: choose an action, call a tool, observe result, continue

But this definition is now too small.

Why:

  • Tool calling alone does not make the system useful.
  • A system can call tools badly.
  • The real question is whether it has the right context, permissions, feedback, and stopping criteria.

Updated definition:

An agent is a goal-directed AI system that can use tools through a harness, observe outcomes, and continue or stop based on feedback and constraints.

#Tool Calling: Tiny Pieces

Tool calling means the model does not only produce prose. It can request an external operation.

Basic loop:

1. Model reads goal + context.
2. Model chooses a tool.
3. Harness validates whether tool use is allowed.
4. Tool executes.
5. Result returns as observation.
6. Model decides next step.
7. Loop continues until done / blocked / stopped.

#The Four Primitive Agent Tools

The most basic tools are not search.

The four primitive tools are:

read
write
edit
delete

Why these four matter:

  • They define whether an agent can perceive and change a workspace.
  • Search is usually a specialized form of read.
  • Shell execution is usually a powerful composition of read/write/edit/delete plus process execution.
  • API calls usually read or mutate external state.

#Explain Each Primitive

Read

The agent can inspect state.

Examples:

  • read files
  • read logs
  • read database rows
  • read web pages
  • read API responses
  • read previous decisions

Risk:

  • wrong source
  • stale source
  • incomplete source
  • private data exposure

Write

The agent can create new state.

Examples:

  • create a file
  • draft a report
  • create a ticket
  • generate a database record
  • create a Google Doc

Risk:

  • creates noise
  • creates misleading artifacts
  • creates state people mistake as approved

Edit

The agent can modify existing state.

Examples:

  • patch code
  • update a spreadsheet
  • revise a document
  • modify a config
  • change a workflow rule

Risk:

  • silent regression
  • loss of intent
  • breaking downstream assumptions

Delete

The agent can remove state.

Examples:

  • delete files
  • remove records
  • close tickets
  • revoke entries
  • clean temporary outputs

Risk:

  • destructive action
  • irreversible loss
  • governance issue

#Tool Risk Ladder

Draw:

read < write < edit < delete
low risk                high risk

Then add:

local sandbox < shared workspace < production system < regulated system

Key teaching point:

The same tool is safe or unsafe depending on where it acts.

Reading a local test file is low risk. Reading PHI, payroll, credentials, or private messages is high risk.

Editing a local draft is low risk. Editing production config is high risk.

Deleting a temp file is low risk. Deleting customer records is existential risk.

#Harness Responsibilities

The harness decides:

  • What context does the model see?
  • What thinking process or task decomposition should the model follow?
  • What tools exist?
  • Which tools require approval?
  • What workspace can the agent touch?
  • What memory persists?
  • What guardrails validate input, output, or tool behavior?
  • What output format is required?
  • What counts as done?
  • What tests or review gates run?
  • When should the agent stop?
  • What gets logged for audit?

  • no ownership

#Gap 1: What CI/CD Is

#One-Sentence Definition

CI/CD is the practice of automatically checking, testing, building, and sometimes deploying software whenever code changes.

#Break The Acronym

CI = Continuous Integration

Goal:

  • make sure new code can safely integrate with the existing codebase

Usually runs:

  • lint
  • typecheck
  • unit tests
  • integration tests
  • build

Question CI answers:

Did this code change break the project?

CD = Continuous Delivery / Continuous Deployment

Continuous Delivery:

  • code is automatically prepared for release
  • a human may still approve deployment

Continuous Deployment:

  • code is automatically deployed after passing checks
  • no manual approval is required, or approval is minimal

Question CD answers:

Can this code safely reach users or a production-like environment?

#Whiteboard Version

code change
   ↓
CI checks: lint / typecheck / tests / build
   ↓
review
   ↓
CD: deploy to preview / staging / production

#Why AI Builders Need It

AI agents can create lots of changes quickly. CI/CD is one of the main ways to prevent speed from becoming chaos.

Key teaching line:

CI/CD is the automated immune system of a software project.

#Minimum Understanding To Teach

You do not need to know every CI/CD tool.

You need to understand:

  • code changes must be checked
  • checks should run automatically
  • tests should block risky changes
  • deployment should be repeatable
  • CI/CD is part of the harness around AI coding

#Gap 2: Relationship Between CI/CD And GitHub Actions

#One-Sentence Definition

GitHub Actions is GitHub's built-in automation system for running CI/CD workflows when repository events happen.

#Relationship

CI/CD = practice / concept
GitHub Actions = one tool that implements it

Other CI/CD tools exist:

  • GitLab CI
  • CircleCI
  • Jenkins
  • Buildkite
  • Cloud Build
  • Vercel / Netlify preview deploy systems

#GitHub Actions Mental Model

event -> workflow -> jobs -> steps -> commands

Definitions:

  • event: something happens, such as push or pull request
  • workflow: YAML file under .github/workflows/
  • job: a group of steps running on a machine
  • step: one command or action
  • runner: the machine executing the job

#Tiny Example

name: CI

on:
  pull_request:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run lint
      - run: npm test
      - run: npm run build

Explain this simply:

When someone opens a PR or pushes to main, GitHub rents a clean machine, checks out the code, installs dependencies, runs lint, runs tests, and tries to build.

#AI Builder Lens

GitHub Actions lets an AI Builder create automatic review gates:

  • "Does this code build?"
  • "Do tests pass?"
  • "Did formatting break?"
  • "Can we deploy a preview?"
  • "Can a reviewer inspect the result?"

This turns AI coding from "agent generated code" into "agent generated code that must survive a shared review process."

#Gap 3: How To Perform Code Reviews And Tests In A Web Environment

#One-Sentence Definition

Web code review means checking not only whether code looks correct, but whether the application actually behaves correctly in a browser.

#Review Layers

Draw this stack:

1. Diff review
2. Static checks
3. Unit tests
4. Integration tests
5. Build check
6. Browser / E2E test
7. Preview deployment review
8. Production monitoring

#Layer 1: Diff Review

Ask:

  • What files changed?
  • Is the change scoped?
  • Does the code match the product intent?
  • Are there risky side effects?
  • Did the agent touch unrelated files?

AI Builder question:

Should this PR exist?

Source anchor:

  • Day 19: AI can check code, but humans must decide whether a PR should exist.

#Layer 2: Static Checks

Usually:

  • formatter
  • linter
  • typechecker

Examples:

npm run lint
npm run typecheck

Purpose:

  • catch obvious code quality and type errors before running the app

#Layer 3: Unit Tests

Unit tests check one small function/component in isolation.

Example question:

Does this formatter function return the right label?

Command might be:

npm test

#Layer 4: Integration Tests

Integration tests check whether multiple pieces work together.

Example question:

Does the frontend correctly call the backend endpoint and render the returned data?

#Layer 5: Build Check

Build check answers:

Can this app compile into something deployable?

Command might be:

npm run build

#Layer 6: Browser / E2E Testing

E2E means end-to-end.

It checks the app the way a user experiences it.

Example questions:

  • Can the user open the page?
  • Can they click the button?
  • Does the form submit?
  • Does the table update?
  • Does the UI break on mobile?

Tool examples:

  • Playwright
  • Cypress

Command might be:

npx playwright test

#Layer 7: Preview Deployment

Preview deployment means:

Every PR gets a temporary live URL so reviewers can inspect the actual app.

Common in:

  • Vercel
  • Netlify
  • Cloudflare Pages
  • GitHub Pages workflows
  • internal staging systems

Review asks:

  • Does it load?
  • Does it look right?
  • Does the flow work?
  • Does it work on mobile?
  • Are errors visible in console/logs?

#Layer 8: Production Monitoring

After deploy, ask:

  • are users seeing errors?
  • are logs clean?
  • is latency acceptable?
  • did conversion/workflow metrics change?
  • did any integration fail?

#Minimal Web Review Checklist

Use this when reviewing AI-generated web changes:

1. What was the product intent?
2. What files changed?
3. Did the agent change unrelated files?
4. Do lint/type/build checks pass?
5. Are there tests for the changed behavior?
6. Did I open the app in a browser?
7. Did I test the core user flow?
8. Did I test mobile/responsive behavior if relevant?
9. Did I inspect console/log errors?
10. Is this safe to merge/deploy?

#AI Builder Lens

Web review is where the abstract AI system touches human experience.

Key teaching line:

Code review asks whether the code is sane. Browser review asks whether the user reality is sane.

AI Builders are not people who know every tool. They are people who can stand at the edge of unclear reality, find the missing context, draw the border of AI capability, and turn that border into a useful product or workflow.
